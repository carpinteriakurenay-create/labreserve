## 二、实现计划

### 执行顺序

```

                                                  │
M2-2 DTO（2 个文件，无依赖）──┘                   │

                        GlobalExceptionHandler（1 行修改）
```

### 关键设计决策

| 决策                                                                                                                                   | 理由                                                                   |
| -------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| **managerName 用两趟法**：先 `selectPage` 查 labs，再收集 managerId 用 `userMapper.selectBatchIds` 批量查 users，构建 id→realName 映射 | 项目现有 Mapper 零自定义 SQL，保持一致性                               |
| **LabHours 全量替换**：PUT 先 `delete` 旧数据，再 for 循环 `insert` 新数据，@Transactional 保证原子性                                  | 每 lab 通常 5-14 条，循环足够；BaseMapper 无 batch insert              |
| **toggleStatus 逻辑**：AVAILABLE → CLOSED，CLOSED/MAINTENANCE → AVAILABLE                                                              | MAINTENANCE 只能通过 update 接口显式设置，不参与 toggle                |
| Controller 不用 class 级 @PreAuthorize，改用 method 级                                                                                 | 读接口（list、getById、getHours）对所有认证用户开放，写接口 ADMIN only |
| `LabStatus` 枚举在 list 接口中由 String 参数转换，无效值抛 `INVALID_STATUS`                                                            | 前端传 status 字符串时需要后端校验                                     |
| `LabCreateRequest` 不暴露 status、equipmentNum                                                                                         | 创建时 status 固定 AVAILABLE，equipmentNum 固定 0                      |
| `GlobalExceptionHandler` 添加 `"NAME_EXISTS"` 到 CONFLICT（409）                                                                       | 与 `USERNAME_EXISTS`、`SERIAL_NUMBER_EXISTS` 语义一致                  |

## 七、M2-7~M2-9 实现计划

### 执行顺序

```
DTO（5 个文件）──> TimeConflictResolver ──> TimeConflictResolverTest (TDD)
                                              │
                                              ├──> BookingService ──> BookingController
                                              │
Booking 错误码已全部在 GlobalExceptionHandler 中映射，无需修改
```

### 关键设计决策

| 决策                                                                                                     | 理由                                                                                                             |
| -------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| **冲突检测三阶段**：LAB_CLOSED → OUTSIDE_OPEN_HOURS → TIME_CONFLICT                                      | lab 关闭时无需继续查 openHours；减少不必要的数据库查询                                                           |
| **approve 时重新 checkConflicts**                                                                        | 防竞态：两笔 PENDING 预约重叠时段，先批一笔后另一笔触发 TIME_CONFLICT                                            |
| **available-slots 用 1 小时间隔**                                                                        | 从 lab hours 生成固定间隔 slots，逐格检查与 approved bookings + courses 的冲突                                   |
| **两趟法填充 joined 字段**                                                                               | 收集 labIds/userIds/approverIds → batch select → Map → 遍历 VO 设置，与 LabService.populateManagerNames 模式一致 |
| **路径顺序**：`/available-slots`、`/mine`、`/pending` 在 `/{bookingId}` 之前                             | Spring 按声明顺序匹配路由，否则 "available-slots" 被当 `{bookingId}` 解析                                        |
| **status 状态机**：PENDING → APPROVED → COMPLETED，PENDING → REJECTED，PENDING/APPROVED → CANCELLED      | 与 api-design.md 一致；仅 PENDING 可修改、仅 APPROVED 可完成                                                     |
| **`TimeConflictResolver` 纯静态、无 Spring 依赖**                                                        | 可独立测试，无 DI 需求                                                                                           |
| **`getCurrentUserId()`**：`(Long) SecurityContextHolder.getContext().getAuthentication().getPrincipal()` | 与 AuthService 一致，由 JwtAuthenticationFilter 在 filter 阶段写入                                               |

## 八、M2-7~M2-9 验证方案

```bash
# 获取 token
ADMIN_LOGIN=$(curl -s -X POST localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"devadmin","password":"admin123"}')

# 注册 STUDENT 用户
curl -s -X POST localhost:8080/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"username":"stu001","password":"pass123","realName":"Student One"}'
# → 201

STUDENT_LOGIN=$(curl -s -X POST localhost:8080/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"stu001","password":"pass123"}')
STUDENT_TOKEN=$(echo "$STUDENT_LOGIN" | sed 's/.*"token":"\([^"]*\)".*/\1/')

# ===== 预约流程 =====

# 1. 查看可用时段
curl -s -X GET "localhost:8080/api/bookings/available-slots?labId=1&date=2026-07-01" \
  -H "Authorization: Bearer $STUDENT_TOKEN"
# → 200 {"code":"SUCCESS","data":[{"startTime":"08:00","endTime":"09:00","available":true},...]}

# 2. 提交预约 → 201
curl -s -X POST localhost:8080/api/bookings \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $STUDENT_TOKEN" \
  -d '{"labId":1,"date":"2026-07-01","startTime":"08:00","endTime":"10:00","purpose":"实验","personCount":2}'
# → 201 {"code":"SUCCESS","message":"预约提交成功","data":{"id":1,"status":"PENDING",...}}

# 3. 我的预约
curl -s -X GET "localhost:8080/api/bookings/mine" \
  -H "Authorization: Bearer $STUDENT_TOKEN"
# → 200 返回 1 条 PENDING 预约

# 4. 冲突时段预约 → 409
curl -s -X POST localhost:8080/api/bookings \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $STUDENT_TOKEN" \
  -d '{"labId":1,"date":"2026-07-01","startTime":"09:00","endTime":"11:00","purpose":"冲突测试","personCount":1}'
# → 409 {"code":"TIME_CONFLICT","message":"预约时间与已有安排冲突","details":null}

# 5. 超出开放时间 → 400
curl -s -X POST localhost:8080/api/bookings \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $STUDENT_TOKEN" \
  -d '{"labId":1,"date":"2026-07-01","startTime":"23:00","endTime":"23:59","purpose":"深夜","personCount":1}'
# → 400 含 OUTSIDE_OPEN_HOURS

# 6. 待审批列表（TEACHER/ADMIN）
curl -s -X GET "localhost:8080/api/bookings/pending" \
  -H "Authorization: Bearer $ADMIN_TOKEN"
# → 200 返回 1 条 PENDING 预约

# 7. 审批通过
curl -s -X PUT localhost:8080/api/bookings/1/approve \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -d '{"approved":true}'
# → 200 {"code":"SUCCESS","message":"审批通过","data":{"status":"APPROVED",...}}

# 8. 重复审批 → 409
curl -s -X PUT localhost:8080/api/bookings/1/approve \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -d '{"approved":true}'
# → 409 {"code":"ALREADY_PROCESSED","message":"该预约已被处理","details":null}

# 9. 修改已审批预约 → 400
curl -s -X PUT localhost:8080/api/bookings/1 \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $STUDENT_TOKEN" \
  -d '{"purpose":"修改测试"}'
# → 400 {"code":"BOOKING_NOT_EDITABLE","message":"只有待审批状态的预约可以修改","details":null}

# 10. 取消预约
curl -s -X PUT localhost:8080/api/bookings/1/cancel \
  -H "Authorization: Bearer $STUDENT_TOKEN"
# → 200 {"code":"SUCCESS","message":"取消成功","data":null}

# 11. 非本人取消 → 403
curl -s -X PUT localhost:8080/api/bookings/1/cancel \
  -H "Authorization: Bearer $ADMIN_TOKEN"
# → 403 {"code":"FORBIDDEN","message":"只能取消自己的预约","details":null}

# 12. 提交新预约 + 审批 + 完成
curl -s -X POST localhost:8080/api/bookings \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $STUDENT_TOKEN" \
  -d '{"labId":1,"date":"2026-07-01","startTime":"14:00","endTime":"16:00","purpose":"完成测试","personCount":1}'
# → 201

curl -s -X PUT localhost:8080/api/bookings/2/approve \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -d '{"approved":true}'
# → 200 APPROVED

curl -s -X PUT localhost:8080/api/bookings/2/complete \
  -H "Authorization: Bearer $ADMIN_TOKEN"
# → 200 {"code":"SUCCESS","message":"确认完成","data":null}

# 12. 未登录 → 403
curl -s -X GET localhost:8080/api/bookings
# → 403 (SecurityConfig 拦截)

# 13. 非 TEACHER/ADMIN 查看待审批 → 403
curl -s -X GET "localhost:8080/api/bookings/pending" \
  -H "Authorization: Bearer $STUDENT_TOKEN"
# → 403
```
