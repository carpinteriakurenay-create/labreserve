### M4-11：审批分布式锁

| 操作 | 文件                                                                        |
| ---- | --------------------------------------------------------------------------- |
| 修改 | `apps/api/src/main/java/com/labreserve/service/BookingService.java`         |
| 修改 | `apps/api/src/main/java/com/labreserve/service/BorrowService.java`          |
| 修改 | `apps/api/src/main/java/com/labreserve/handler/GlobalExceptionHandler.java` |

两个 Service 均：

- 构造函数注入 `RedissonClient`
- `approve` 方法包裹 `tryLock(5, TimeUnit.SECONDS)`，失败抛 `BusinessException("LOCK_FAILED", "...")`
- `finally` 中 `lock.unlock()` 确保锁释放
- 无 `leaseTime`，依赖 Redisson watchdog 自动续期（默认 30s TTL，每 10s 续期）

锁粒度：

| 方法             | 锁 Key              |
| ---------------- | ------------------- |
| `approveBooking` | `booking:lock:{id}` |
| `approveBorrow`  | `borrow:lock:{id}`  |

GlobalExceptionHandler 新增：`LOCK_FAILED` → HTTP 409 Conflict。

## 二、实现计划（M4-9 ~ M4-11）

### 执行顺序

```
M4-9 (pom + yml + RedisCacheConfig + @EnableCaching)
  │
  └──> M4-10 (UserInfo fix → LabService → UserService → DashboardService)
  │
  └──> M4-11 (BookingService → BorrowService → GlobalExceptionHandler)
```

M4-9 是基础设施，必须先完成。M4-10 和 M4-11 可并行，但都依赖 M4-9 的 `RedissonClient` Bean 就绪。

### 关键设计决策

| 决策                                                | 理由                                                                      |
| --------------------------------------------------- | ------------------------------------------------------------------------- |
| Redisson 替代 Lettuce 作为 Redis 客户端             | 同时提供 `RedissonSpringCacheManager` 和 `RLock` 分布式锁，统一一个客户端 |
| `listLabs` / `listUsers` 不缓存                     | `IPage` 序列化问题 + 分页参数多变命中率低                                 |
| 锁等待 5s，无 leaseTime（watchdog 自动续期）        | 5s 等不到即抛错给用户；watchdog 防长事务锁提前过期                        |
| `createLab` / `createUser` 使用 `allEntries = true` | 新建后列表总条目变化，全部缓存失效最简单                                  |
| `UserInfo` 补 `@NoArgsConstructor`                  | Jackson 反序列化必须有无参构造器，`@Builder` 不生成                       |
| `@CacheEvict` 在 `@Transactional` 外层              | Spring 默认先执行方法再清缓存；事务回滚时缓存可能已清（TTL 窗口内可接受） |
| `LOCK_FAILED` → 409                                 | 与 `TIME_CONFLICT`、`ALREADY_PROCESSED` 一致，语义为资源冲突              |

### 分布式锁验证

```bash
# 快速连续发送两次审批请求（同一 booking）
curl -s -X PUT "localhost:8080/api/bookings/1/approve" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"approved":true}' &
curl -s -X PUT "localhost:8080/api/bookings/1/approve" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"approved":true}' &

wait
# 第一个 → 200（审批成功）
# 第二个 → 409 {"code":"LOCK_FAILED","message":"该预约正在被其他管理员处理，请稍后重试"}

```

## 七、关键设计决策

### 自定义安全注解

JwtAuthenticationFilter 将 `principal` 设置为 `Long(userId)`，但 Spring Security Test 的 `@WithMockUser` 使用 `String` principal。Controller 中 `getCurrentUserId()` 直接 `(Long) getPrincipal()` 会抛 `ClassCastException`。

**方案**：创建 `@WithMockStudent`/`@WithMockTeacher`/`@WithMockAdmin` 三个自定义注解，每个配备 `WithSecurityContextFactory` 实现，设置 `new UsernamePasswordAuthenticationToken(userId, null, List.of(new SimpleGrantedAuthority("ROLE_XXX")))`。

### Redisson 测试隔离

`RedisCacheConfig.cacheManager(RedissonClient)` 在测试环境找不到 Redis 时会抛连接异常。`BookingService.approve()` 和 `BorrowService.approve()` 使用 `redissonClient.getLock()`，MockMvc 请求进入真实 Service 时 null RedissonClient 会 NPE。

**方案**：创建 `RedisTestConfig`（`@TestConfiguration`），提供 `@Primary RedissonClient` mock + 简易 `PasswordEncoder`。通过 `@Import(RedisTestConfig.class)` 注入测试上下文，配合 `spring.main.allow-bean-definition-overriding=true` 覆盖真实 Bean。

### H2 枚举列类型

MyBatis-Plus 默认使用 `Enum.name()` 序列化枚举为 `VARCHAR`，但 H2 schema 初始设计为 `TINYINT`（ordinal），导致 `No enum constant com.labreserve.enums.LabStatus.0`。

**方案**：将所有枚举列从 `TINYINT` 改为 `VARCHAR(20)`，与 MySQL 生产库一致。data.sql 中的枚举值使用名称字符串（如 `'AVAILABLE'`、`'PENDING'`）。

### 测试数据隔离

每个 `@SpringBootTest` 测试类独立加载 ApplicationContext，`@Sql(executionPhase = BEFORE_TEST_METHOD)` 在每个 `@Test` 前执行 DROP + CREATE + INSERT，确保测试间数据隔离。
