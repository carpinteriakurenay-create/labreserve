## 一、技术风险

### R-T01: 时间冲突检测算法实现复杂度高

| 维度       | 评估                                              |
| ---------- | ------------------------------------------------- |
| **概率**   | 高 (60%) — 新人/团队首次实现，易遗漏边界 case     |
| **影响**   | 严重 — 核心功能，做错会导致重复预约，直接不可用   |
| **涉及**   | F-14 (审批)、F-20 (冲突检测)、F-13 (可用时段查询) |
| **优先级** | P0                                                |

**问题描述：**

冲突检测需要将三类数据取交集：

1. 实验室开放时段（`lab_hours`，每周规则）
2. 已审批的预约（`bookings`，具体日期）
3. 周期性课程安排（`courses`，周几 + 日期范围）

交集逻辑的边界 case 包括：跨天时段、同一实验室同一时段多个开放窗口、预约和课程之间的 1 分钟重叠、课程只在特定周范围有效等。SQL 层面只能做粗筛（`lab_id + date/day_of_week`），精确的重叠判断在 Java 代码中完成。

**关键测试场景：**

| 场景                                             | 预期            |
| ------------------------------------------------ | --------------- |
| 完全重叠（14:00-16:00 预约 vs 14:00-16:00 课程） | 冲突            |
| 部分重叠（14:00-16:00 vs 15:00-17:00）           | 冲突            |
| 首尾相连（14:00-16:00 vs 16:00-18:00）           | 不冲突          |
| 包含关系（14:00-18:00 vs 15:00-16:00）           | 冲突            |
| 跨天时段（22:00-02:00）                          | 抛异常 / 不支持 |

**应对策略：**

- **预防**：将冲突检测逻辑封装为独立纯函数 `TimeConflictResolver`，单独编写 JUnit 5 参数化测试覆盖所有重叠场景；审批前再次实时查询数据库做最终校验
- **缓解**：审批接口返回 `TIME_CONFLICT` 时附带冲突详情（哪条预约/课程、时间段），方便用户理解
- **应急**：若缺陷逃逸到生产，管理员可手动取消冲突预约，通过数据库直查修复

### R-T02: Spring Security + JWT 集成链配置复杂

| 维度       | 评估                                          |
| ---------- | --------------------------------------------- |
| **概率**   | 中 (40%)                                      |
| **影响**   | 严重 — 认证失败 = 全站不可用                  |
| **涉及**   | F-02 (登录)、F-03 (获取用户)、F-05 (权限控制) |
| **优先级** | P0                                            |

**问题描述：**

当前 `pom.xml` 缺少 `spring-boot-starter-security` 和 `jjwt` 依赖。Spring Security 6.x 的 Filter Chain DSL 与旧版本差异大，常见踩坑点：

- `SecurityFilterChain` Bean 配置不当导致所有请求 401 或所有请求放行
- `OncePerRequestFilter` 中异常处理不当时，Spring Security 的 `ExceptionTranslationFilter` 不会生效
- `@PreAuthorize("hasRole('ADMIN')")` 与 JWT 中 role 字段的映射需要自定义 `GrantedAuthority` 前缀
- CORS 配置与 Spring Security 的 CORS 配置同时生效时可能冲突

**应对策略：**

- **预防**：Day 1 优先实现认证模块并通过集成测试；参考 Spring Security 6 官方 migration guide 的 Filter Chain 写法；JwtAuthenticationFilter 实现后立即用 `@WebMvcTest` + `@WithMockUser` 覆盖角色场景
- **缓解**：在 `SecurityConfig` 中配置 `/api/auth/**`、`/api/health`、`/actuator/health` 为公开路径，即使 filter 链出错也能保证登录可用
- **应急**：若生产出现 401 误拦，临时通过环境变量 `SECURITY_DEBUG=true` 开启 Spring Security debug 日志排查

### R-T03: Cursor-based 分页与 Element Plus 分页组件不兼容

| 维度       | 评估                                    |
| ---------- | --------------------------------------- |
| **概率**   | 中 (50%)                                |
| **影响**   | 中等 — 分页体验受影响，但不阻塞核心功能 |
| **涉及**   | 全部列表接口                            |
| **优先级** | P1                                      |

**问题描述：**

Element Plus 的 `ElPagination` 组件是 offset-based 设计（`page` + `pageSize` + `total`），不支持 cursor-based 的"加载更多"交互。需要二选一：

1. 放弃 `ElPagination`，使用 `ElButton` + `v-infinite-scroll` 实现加载更多
2. 封装一个适配层：将 cursor 映射为 page（但会丢失 cursor 分页的稳定性优势）

**应对策略：**

- **预防**：Day 1 先封装一个 `useCursorPagination` composable，实现 `loadMore()` + `reset()` 方法，统一列表页的数据获取和追加逻辑
- **缓解**：列表数据量不大（< 200 条）的场景（如我的预约、我的借用），后端提供可选的 `total` 字段，前端可渲染"已显示 X/总共 Y"
- **应急**：若 cursor 方案在联调中证明过于复杂，P1 阶段可切换部分低频列表接口为传统 offset 分页

### R-T04: Redis 分布式锁在审批环节的可靠性

| 维度       | 评估                                                        |
| ---------- | ----------------------------------------------------------- |
| **概率**   | 低 (20%)                                                    |
| **影响**   | 中等 — Redis 不可用时审批功能降级，但可通过数据库乐观锁兜底 |
| **涉及**   | F-14 (审批)                                                 |
| **优先级** | P0                                                          |

**问题描述：**

Redisson 依赖尚未引入 `pom.xml`。分布式锁的成熟度较高，但两个风险点存在：

1. **锁超时**：若审批逻辑耗时超过 Redisson watchdog 默认 30s（极少发生），锁自动释放导致双重审批
2. **Redis 不可用**：锁获取失败时，审批直接报错，阻塞所有审批操作

**应对策略：**

- **预防**：Redisson 配置 `tryLock(0, 10, TimeUnit.SECONDS)`，watchdog 自动续期；审批逻辑耗时远小于 10s（仅为 SELECT + UPDATE）
- **缓解**：数据库层面加 `UNIQUE(booking_id, approver_id)` 或版本号字段，防止重复审批写入
- **应急**：Redis 不可用时降级为数据库悲观锁（`SELECT ... FOR UPDATE`），虽然并发性差但保证数据正确性；监控 Redis 可用性并配置告警

### R-T05: 14 张表的软删除 + 唯一约束冲突

| 维度       | 评估                                                          |
| ---------- | ------------------------------------------------------------- |
| **概率**   | 中 (30%)                                                      |
| **影响**   | 中等 — 影响特定操作（重新注册同名用户、重新登记同序列号设备） |
| **涉及**   | users.username、equipment.serial_number                       |
| **优先级** | P1                                                            |

**问题描述：**

数据模型设计文档已指出此风险（data-model.md 第 269-271 行）：MySQL 8.0 不支持条件唯一索引，软删除的记录仍然占用唯一约束。如果用户"张三"被软删除后，管理员想重新创建同名用户，会触发 `Duplicate entry` 异常。

**应对策略：**

- **预防**：软删除时对唯一字段追加时间戳后缀（如 `zhangsan_del_1719657600`），释放原值供新记录使用（已在 data-model.md 记录此方案）
- **缓解**：在 `UserServiceImpl.delete()` 和 `EquipmentServiceImpl.delete()` 中统一实现此逻辑
- **应急**：Admin 可手动修改已删除记录的用户名/序列号后再创建；或通过 `ON DUPLICATE KEY UPDATE` 复用已删除的记录

## 二、依赖风险

### R-D01: MySQL 数据库不可用

| 维度       | 评估                                       |
| ---------- | ------------------------------------------ |
| **概率**   | 低 (10%) — Docker 容器有健康检查和自动重启 |
| **影响**   | 严重 — 全站不可用，所有接口依赖数据库      |
| **优先级** | P0                                         |

**问题描述：**

MySQL 是唯一的持久化存储。Docker 容器挂掉、宿主机磁盘满、连接池耗尽都可能造成不可用。

**应对策略：**

- **预防**：`docker-compose.yml` 已配置 `restart: unless-stopped` + `healthcheck`；生产环境使用云数据库（自动备份 + 故障切换）；后端配置 HikariCP 连接池超时 `connection-timeout: 5000`，避免线程饥饿
- **检测**：`/api/health` 包含数据库连接检查；Actuator `health` 端点暴露 `db` 状态
- **应急**：MySQL 恢复前，后端返回 503 + "服务暂时不可用"，前端展示维护页面；Docker 环境下执行 `docker compose restart mysql`

---

### R-D02: Redis 不可用

| 维度       | 评估                                                                 |
| ---------- | -------------------------------------------------------------------- |
| **概率**   | 低 (10%)                                                             |
| **影响**   | 中等 — 缓存未命中导致性能下降（DB 直接承担读压力），审批分布式锁降级 |
| **优先级** | P1                                                                   |

**问题描述：**

Redis 承担两个角色：缓存（@Cacheable）和分布式锁（Redisson）。缓存不可用是降级而非中断（数据库兜底），但分布式锁不可用会阻塞审批（详见 R-T04）。

**应对策略：**

- **预防**：所有 `@Cacheable` 方法必须有 try-catch，Redis 异常时跳过缓存直接查库；Redisson 获取锁失败时降级为数据库锁
- **检测**：Actuator `health` 端点暴露 `redis` 状态
- **应急**：若 Redis 长时间不可用，通过环境变量 `REDIS_ENABLED=false` 全局禁用 Redis，所有功能降级运行直到 Redis 恢复

### R-D03: JWT Secret 泄露

| 维度       | 评估                                                |
| ---------- | --------------------------------------------------- |
| **概率**   | 低 (15%) — 前提是代码仓库权限管理规范               |
| **影响**   | 严重 — 攻击者可伪造任意用户的 token，获得管理员权限 |
| **优先级** | P0                                                  |

**问题描述：**

JWT 的安全性完全依赖于 secret 的保密性。若 secret 硬编码在 `application.yml` 中且提交到 Git，任何人拿到源码即可签发合法 token。

**应对策略：**

- **预防**：
  - `application.yml` 中的 JWT secret 使用占位符 `${JWT_SECRET}`，实际值通过环境变量注入
  - `.env` 文件加入 `.gitignore`（已在 `.gitignore` 中）
  - CI 中通过 GitHub Secrets 注入
  - 生产环境使用 256-bit 随机密钥（`openssl rand -base64 32`）
- **检测**：GitHub Secret Scanning 配置检测 JWT secret 模式
- **应急**：若怀疑泄露，立即轮换 JWT secret（所有已签发 token 失效，用户需重新登录）

### R-D04: Element Plus 大版本升级导致的破坏性变更

| 维度       | 评估                              |
| ---------- | --------------------------------- |
| **概率**   | 低 (20%)                          |
| **影响**   | 低 — 锁定版本后仅在主动升级时触发 |

## 四、安全风险

### R-SEC01: JWT Token 无失效机制

| 维度         | 评估                                                  |
| ------------ | ----------------------------------------------------- |
| **概率**     | 中 (40%) — 用户主动退出后期望 token 立即失效          |
| **影响**     | 中等 — 已登出用户的 token 仍可被重放，直到 24h 过期   |
| **风险类型** | 会话管理                                              |
| **OWASP**    | A07:2021 — Identification and Authentication Failures |

**问题描述：**

JWT 无状态意味着签发后无法主动撤销。用户点击"退出登录"只是前端清除了 token，后端仍然接受该 token。如果 token 在过期前被窃取，攻击者可持续使用。

**应对策略：**

- **预防**：
  - Token 有效期设为 24h（满足"一天内使用"需求，限制暴露窗口）
  - 实现 Redis token 黑名单：登出时将 `jti`（JWT ID）写入 Redis，TTL = token 剩余有效时间；JwtAuthenticationFilter 验证时先查黑名单
  - Access Token 短期化（如 15min）+ Refresh Token（24h）是最佳实践，但 MVP 先不做
- **检测**：登录时生成 UUID 作为 `jti` claim，日志记录登录/登出事件
- **应急**：若发现 token 被滥用，轮换 JWT secret 使所有 token 立即失效

---

### R-SEC02: 密码策略过弱

| 维度         | 评估                                            |
| ------------ | ----------------------------------------------- |
| **概率**     | 高 (80%) — 学生默认使用简单密码（学号、123456） |
| **影响**     | 中等 — 弱密码容易被暴力破解或撞库               |
| **风险类型** | 认证安全                                        |
| **OWASP**    | A07:2021                                        |

**问题描述：**

当前注册接口仅要求密码 length ≥ 6，不强制复杂度。BCrypt 能防彩虹表但防不了弱密码（`123456` 的 BCrypt hash 仍是弱密码的 hash）。

**应对策略：**

- **预防**：
  - 注册接口增加密码复杂度要求：至少 8 位、包含字母和数字
  - 后端 `@Pattern` 校验 + 前端 Element Plus 表单实时强度提示
  - 登录接口限流：同一 IP / 同一账号连续失败 5 次后锁定 15 分钟（Redis 计数器实现）
- **检测**：管理后台展示弱密码用户列表（线下通知修改）
- **应急**：若发现撞库迹象，管理员批量禁用可疑账号 (`PUT /users/{id}` enabled=false)

---

### R-SEC03: 未授权的预约/借用操作

| 维度         | 评估                                      |
| ------------ | ----------------------------------------- |
| **概率**     | 中 (35%)                                  |
| **影响**     | 严重 — 学生越权取消他人预约、查看他人数据 |
| **风险类型** | 访问控制                                  |
| **OWASP**    | A01:2021 — Broken Access Control          |

**问题描述：**

关键是"用户只能操作自己的资源"这个约束是否在每个接口都生效。例如：

- `PUT /api/bookings/{bookingId}/cancel` — 是否校验 `booking.userId == currentUserId`？
- `GET /api/reviews?userId=X` — 是否可以查看其他用户的评价？
- `PUT /api/bookings/{bookingId}/approve` — 是否校验审批人是 TEACHER 或 ADMIN？

仅靠前端隐藏按钮不够，攻击者可直接发 HTTP 请求。

**应对策略：**

- **预防**：
  - Service 层每个写操作校验归属：`booking.getUserId().equals(currentUserId)` 或抛 `FORBIDDEN`
  - 审批操作校验角色：`@PreAuthorize("hasAnyRole('TEACHER', 'ADMIN')")`
  - Controller 层不信任前端传来的 `userId`，改用 `SecurityContextHolder` 获取当前用户
  - 单元测试必须覆盖"他人资源不可操作"的 case
- **检测**：安全测试 checklist — 用学生 A 的 token 操作学生 B 的预约/借用/评价，预期 403
- **应急**：审计日志记录每次取消/修改的操作人和 IP，事后可追溯

### R-SEC04: SQL 注入 — 动态排序/筛选参数

| 维度         | 评估                                           |
| ------------ | ---------------------------------------------- |
| **概率**     | 低 (20%) — MyBatis-Plus 参数化查询是默认安全的 |
| **影响**     | 严重 — 数据泄露或破坏                          |
| **风险类型** | 注入攻击                                       |
| **OWASP**    | A03:2021 — Injection                           |

**问题描述：**

MyBatis-Plus 的 `LambdaQueryWrapper` 使用参数化查询，绝大多数场景安全。但两个例外：

1. **动态排序**：`sortBy` 和 `sortOrder` 参数如果直接拼接到 SQL `ORDER BY` 子句中（MyBatis-Plus 的 `last()` 方法），可能被注入
2. **自定义 SQL**：手写 mapper XML 中使用 `${}` 而非 `#{}`

**应对策略：**

- **预防**：
  - `sortBy` 参数在 Controller 层白名单校验：只允许 `["createdAt", "updatedAt", "date", "name"]` 等预定义字段
  - `sortOrder` 校验：只允许 `"asc"` 或 `"desc"`
  - Mapper XML 中统一使用 `#{}` 占位符，`${}` 仅为表名/列名动态拼接且在 Java 代码中白名单过滤
  - MyBatis-Plus 3.5 的 `orderBy(true, condition, column)` 方法天然防注入（参数化）
- **检测**：Code Review checklist — 每个 Mapper XML 检查无 `${}` 拼接用户输入

### R-SEC05: XSS — 通知/评价/预约用途为富文本输入点

| 维度         | 评估                              |
| ------------ | --------------------------------- |
| **概率**     | 中 (30%)                          |
| **影响**     | 中等 — 恶意脚本在查看者浏览器执行 |
| **风险类型** | 跨站脚本                          |
| **OWASP**    | A03:2021                          |

**问题描述：**

以下字段是用户的自由文本输入，且会在其他用户页面展示：

- `notices.content` — 管理员发布，所有用户可见
- `reviews.comment` — 学生评价，在实验室详情页展示
- `bookings.purpose` — 申请用途，审批人可见

如果后端不做转义，`<script>alert('xss')</script>` 会在查看者浏览器执行。

**应对策略：**

- **预防**：
  - 后端响应中 HTML 实体转义（Spring Boot 默认的 Jackson 序列化不自动转义）
  - 前端 `v-text` / `{{ }}` 替代 `v-html`（Vue 默认转义，除非显式用 `v-html`）
  - 若确实需要富文本（通知内容），后端用 OWASP Java HTML Sanitizer 清洗，仅允许安全标签（`<b>`, `<p>`, `<br>`）
- **检测**：安全测试用例 — 在评价/用途字段输入 `<script>alert(1)</script>`，确认不弹窗
- **应急**：若发现 XSS，紧急部署后端 HTML 过滤；通知内容数据库批量清洗

### R-SEC06: CORS 配置过宽

| 维度         | 评估                                                                           |
| ------------ | ------------------------------------------------------------------------------ |
| **概率**     | 中 (40%) — 开发时为了方便常配 `allowedOrigins: "*"`                            |
| **影响**     | 低 — 仅允许 credentialed 请求时 `*` 会报错；但过宽配置允许任意站点发起跨域请求 |
| **风险类型** | 跨域配置                                                                       |
| **OWASP**    | A05:2021 — Security Misconfiguration                                           |

**问题描述：**

开发环境 Vite dev server 在 `localhost:5173`，后端在 `localhost:8080`，需要 CORS。生产环境前后端同域（Nginx 反向代理），不需要 CORS。

常见的错误：将开发环境的 `allowedOrigins: "*"` 带到生产配置中。或者生产环境同时开放了 CORS。

**应对策略：**

- **预防**：
  - dev profile 中 CORS 限为 `http://localhost:5173`
  - staging/production profile 不配置 CORS（Nginx 同域代理）
  - SecurityConfig 中 CORS 配置跟随 Spring profile
- **检测**：`curl -H "Origin: https://evil.com" https://labreserve.univ.edu.cn/api/health` 确认不返回 `Access-Control-Allow-Origin`
